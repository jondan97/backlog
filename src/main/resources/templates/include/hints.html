Existing Projects

All the projects that exist in this application can be found here.

Agile enthusiasts would prefer to use the term product instead of the word project.

A product is something (physical or not), that is created through a development process and that provides value to the customer. A product could be anything, ranging from a software application to a service.

The key part is the fact that the customer and their continuous feedback is valued more than the final product itself and the bureaucracy that it is surrounded by.

An example product, would be something like this:

Title: Bike Application

Description: The customer will be able to use this application in order to rent bikes from stations all over across the city. The application will be an online platform that tracks and manages all the bikes stored in the company’s stations.

Total effort: 170

Developers working: 8

Estimated sprints needed: 12

Description

This is the description of the product.

It is usually written by the product owner, and it describes the vision of what the team is going to do, make or accomplish.

It should be short, and it should give a clear definition of the value it adds to the market.

Developers Working

This is the number of developers working in this product.

The developers are considered a team, and they should have all the skills necessary to take the vision of the product owner and make it a reality. In a team, there are no roles, and everybody (if possible) is involved in all the phases of the development process.

The number of recommended developers ranges from 3 to 9. More than that and it gets complicated for team members to collaborate properly.

If the number of developers is not stable (which is not recommended), you should leave it at 0.

Some notes: The number of developers does not actually impact anything in this application. It was implemented for educational purposes.

Total Effort

This is the total effort of the project.

The total effort is calculated by aggregating all the effort of the tasks and bugs that the project includes. The tasks/bugs can be in any state, either that is in the backlog or finished.

The total effort is used in the calculation of the product burndown chart, where both the effort burned each iteration and the total effort of the project are shown.

Estimated Total Effort

This is the estimated total effort of the project.

The estimated total effort of the project is the aggregation of all the effort estimated for each epic or story without a parent.

In Agile and especially Scrum, it is important to make initial estimations for each item in the backlog without defining the set of tasks and bugs that the items comprise of. This helps in roughly calculating the number of sprints that will be needed as well as in understanding the amount of work that each item requires. The estimated effort for an item may continually change, depending on team discussion and based on the results of the decomposition of the item into smaller ones. Moreover, customer needs can change which will also impact the effort that an item has, and re-estimation of the effort may be required.

Some notes:

  • The user can only estimate the effort of stories and epics.
  • The total effort of each story and epic is the aggregation of the effort of all the tasks and bugs that the story or epic includes.
  • The user can specify the effort needed for each task and bug, but they cannot set the estimated one, as it is only reserved for the parents (epics/stories).

Estimated Sprints Needed

The number of sprints required to finish the product are stated here.

At the beginning, it is quite difficult to accurately predict the estimated sprints needed unless the team has worked in the past together. A number should nevertheless be calculated, as it helps in the product burndown chart where an ideal sprint effort burn is shown, along with the effort burn of the sprints that have already been executed.

The ideal sprint effort burn which is the velocity of each iteration, helps the team with tracking progress and spotting any problems that should be solved before it is too late.

This number always changes (automatically), depending on the team velocity of the previous sprint.

If a prediction is not possible, then this shows 0 and the ideal sprint burn in the project progress page is not calculated.

Some notes:

If the total effort is 0, then the estimated sprints are based on the estimated total effort. If the estimated total effort is also 0, then the value is set to "Not Known Yet".

The formula used to calculate the sprints needed is the following:

  • (Sprints Needed) = (Total Effort) / (Team Velocity)

The team will burn a certain amount of effort (velocity) each iteration, until they burn all the effort included in the project.

Ready Sprint

This is the Ready Sprint.

When this is shown, the sprint is in a ready state.

During the ready state, items can be added, updated or deleted from the sprint backlog. An item can be added to the sprint backlog, by clicking on the “Move to Sprint” button, next to each item, in the product backlog.

The user has the ability to either move a set of items at once (by moving epics/stories), or by moving tasks/bugs individually. When the sprint is initialized however, only tasks and bugs will be taken into account during the task board view.

Active Sprint

This is the Active Sprint.

When this is shown, the sprint is in an active state.

During the active state, items cannot be added or deleted from the sprint. What is included inside the active sprint, stays in the active sprint, until the end of it.

A timer is set, which is based on the duration configured for the sprint. When it expires, all the items that were included in the sprint, will have one of the following behaviours:

  • If the item is done, then it will disappear from the sprint backlog, and it can only be accessed through the sprint history or the project progress.
  • If the item is not done, then it will stay in the sprint backlog of the next ready sprint, with the status that it had, before the active sprint was finished.

If the product owner decides to finish the sprint earlier for whatever reason, he can do so by clicking the “Finish” button.

Sprint Goal

This is the goal that the team is trying to achieve, by the end of the sprint.

A sprint goal shows the desired outcome of an iteration and ensures that everyone is on the same page.

This is usually done by the product owner, but the team may also contribute. After the goal is defined, it should be accepted by the stakeholders and especially the customer.

An example goal, could be something like:

  • Develop the checkout process of the application.
  • Allow teachers to view and update grades.

After the end of the sprint, the team checks on the goal, and compares it with the reality. Based on the results, they self-reflect and thrive for improvement.

Sprint Duration

This is the duration of the sprint.

The duration of the sprint tracks how much the current iteration should last for. After the duration has passed, the sprint expires and it turns from an active state to a ready state, which also includes all the unfinished tasks of the previous sprint.

The sprint duration should be the same for each sprint, as it helps the team forecast future sprint outcomes and in the calculation of a steady speed. If the duration is different for each iteration, then the velocity is not very accurate compared to the velocity of a project with iterations of the same length.

The recommended sprint duration is 2 weeks, but it can reach up to 30 days (4 weeks).

The product owner is also able to finish the sprint earlier than the normal deadline.

Sprint End Date

The day which the sprint will move from active state to ready state.

It will automatically do so, by the end of the day, which means at 23:59:59 of the final day of the sprint.

Velocity

The velocity of the sprint is the number of tasks/bugs finished by the end of the execution of a sprint.

In this application, the team velocity is calculated by summing up the effort of all the tasks and bugs that were finished in the previous sprint.

The sprint velocity helps calculate the team’s speed, which in return, helps in forecasting the future sprints and their results.

It is the key metric in Scrum, as it works as an indicator, which shows whether the team is improving each sprint, or becomes worse. The more the velocity each sprint, the more the team is improving in working together and finishing user stories and requirements.

Velocity is used both in the product burndown chart, and the sprint burndown chart. The former helps the product owner in figuring out how many sprints it will take the team to achieve a desired level of functionality that can be shipped, whereas the last helps the team in tracking their progress though out the active sprint and in detecting or solving any problems.

Some notes: An initial team velocity is requested to be set during the creation of a project, in order to calculate the initial number of estimated sprints needed for the project to finish. The initial velocity can be based on previous sprints of others projects or should be set to 0 if it is unknown and an initial prediction is not required. When the latest sprint finishes execution, its velocity will be calculated and will automatically be set for the velocity of the project.

Sprint Backlog

This is the sprint backlog.

Here are the items that are included in the sprint and which the development team will work on.

Only the development team can decide upon the items they will work on, as they are the ones that better understand the difficulties behind each item, and only they can assess what can be accomplished during a sprint.

The difference between the product backlog and the sprint backlog is the fact that the product backlog contains everything the team might ever work on, whereas the sprint backlog only contains the items that they will work during one sprint.

Once a sprint has started, items cannot be added or removed from the sprint backlog. What was in the sprint backlog at the beginning, will stay in that until the end of the sprint. However, since not all are known at the beginning of the sprint, the ability to ‘refine’ the items included in the sprint has been left in. Should the effort be re-calibrated, then the user can do so by updating the item from the item page.

Once the sprint has finished the items that are done disappear from the sprint backlog (can be accessed from sprint history or project progress), and the items that were left, remain at the state they had before the end of the sprint. In Scrum, there is no such thing as 80% done. Everything that was not done, is added back to the sprint backlog and only the finished items contribute to the sprint’s velocity.

By the end of each sprint, the team is expected to have finished enough tasks, so that a potentially shippable product (or feature), has been created.

Product Backlog

This is the product backlog.

The product backlog is a list of items that the team may deliver in order to add value to a product and fulfil a specific need of the customer. These items include:

  • New features
  • Changes to existing features
  • Bug fixes
  • Infrastructure changes
  • Other activities such as knowledge acquisition

Anyone can add to the backlog, but the essential items come from the product owner as well as the customer and they are sorted from the most important item, to the least important one.

The product backlog is a living document. It always changes and evolves according to the feedback received from the customer. A healthy backlog would always grow instead of shrinking.

One of the major activities that is done within the team in collaboration with the product owner, is something called backlog refinement.

The backlog refinement helps the team in achieving clarity as well as understanding the scope and the vision. The backlog refinement includes the following topics:

  • Estimation of effort
  • Decomposition of large product backlog items (PBIs) into smaller ones
  • Clarification of requirements

Before these topics are explained, the term INVEST concerning items should be understood first. It stands for:

  • Independent: The item dependency should be reduced from one to another. This allows the product owner to create PBIs that are prioritized by business value rather than technical efficiencies.
  • Negotiable: The customer, the product owner as well as the team must work together daily for the maximum potential value to be reached.
  • Valuable: A PBI should be developed around the customer needs, so that the business value is clear and existent.
  • Estimable: If the team is not able to clearly estimate a PBI, then it should be broken into smaller pieces where it is further understood.
  • SmallAn individual PBI should be very small, so that it is doable in the 1/4th of a sprint. Doable includes full testing, integration, refactoring etc.
  • Testable: Full coverage (including edge cases) should be clear and doable amongst the team for each PBI.

The team together with the product owner sit together in a room and discuss the topics formerly stated above.

Estimation of effort

Each item is presented to the team. The team is asked to write down on a piece of paper the level of difficulty for each of the items. In this application, the easiest starts from 1 and ends to 10 which is the hardest. Afterwards, they are required to explain the reasoning behind the number they chose. This is called Planning Poker and helps in forming discussions between the team as well as revealing valuable information that was not clear before to the whole team.

Decomposition of large product PBIs into smaller ones

If something is not clear enough, then the PBI should be broken into a smaller one. This not only helps in clarifying things, but also controlling the amount of work the team will be able to do during a sprint. An Epic, which is the largest item, should be broken down into User Stories and user stories should be broken down into Tasks/Bugs, which will be added to the sprint.

Clarification of Requirements

The product owner should make the requirements of the customer clear and the vision he set for the product. This helps the team understand the product better as a whole and allows them to revolve around adding customer value instead of technical value.

Some notes: The product backlog should NEVER contain stray user stories or tasks. The proper structure a backlog should follow, is something like this: Epics broken into User Stories broken into Tasks/Bugs. However, the developer has left the ability to have stray stories and tasks in the backlog, but it is not recommended.

Item Title

This is the title of the item.

It should be as short as possible, and as clear as possible, for the team to understand what it is about.

A title usually describes in a few words either a feature that will be part of the product or technical aspects such as bugs.

Item Description

This is the description of the item.

The description of the item should directly explain what this item is about. A common method used by Agile enthusiasts in order to describe an item, is the following:

As a I can so that .

However, this is only an example, not a prescription and any method found appropriately during team discussion can be used.

Some notes:

  • A product backlog item is more about the what, whereas a Task or a Bug is more about the how.
  • A well-formed description of a product backlog item represents a distinct business value, whereas a task represents a step from the team to create that value.

Item Acceptance Criteria

This is the acceptance criteria of an item.

The acceptance criteria of an item are the specifications of when the item is considered done.

Acceptance criteria can be the ability to work on a specific platform, or the level of satisfaction and feedback received by the customer.

Usual examples of when an item is considered done include:

  • Properly tested
  • Refactored
  • Potentially Shippable

Some notes: These specifications mainly concern tasks and bugs during a sprint execution, but they can also be used in the definition of a product backlog item and outside sprints.

Item Type

This is the type of an item.

An item can be an Epic, a User Story, a Task or a Bug.

Epics are large bodies of work that can be broken down into many small individual pieces (user stories). They help the team define a structure for the product, as they usually represent huge chunks and core features of the product. They are used to create a hierarchy and they usually require more than one sprint to reach a finished state. This is due to the fact that by the end of each sprint, customer feedback may add more items or new needs may be discovered.

Stories are in a way, software system requirements but from the perspective of the customer. They are formed using language that the end user understands, without any technical slangs and they provide context for the team to better understand the customer and realize the reasons behind what the team is building or the value these stories create. Stories are a few sentences in colloquial language that describe the desired outcome of a feature.

Tasks/Bugs are the smallest unit of work possible. They are created during the backlog refinement phase, where the team discusses in technical detail, how each story should be handled and solved. Tasks/Bugs are a set of technical descriptions that only concern the team and provide the ability to clearly see how each software requirement will be developed, tested, integrated etc. These are the only types of item that count in a sprint, and the calculation of the sprint velocity.

Item Parent

An epic can only contain user stories.

A user story can only contain tasks or bugs.

A task or a bug are the smallest item possible and cannot contain any other form of item.

It is recommended that only epics are included in the backlog, which in return contain user stories that have their own tasks and bugs.

However, the developer has left the ability to contain stray stories and tasks/bugs in the backlog, in case they want to be saved for editing later during the product development.

Item Priority

This is the priority of an item.

It describes how important an item is. This mainly concerns the product backlog, as it is sorted from the most important item to the least important one. This clarifies to the team, which of the product backlog items, are the most essential to complete and the most important from the perspective of the customer. Finishing important items first, adds customer value to the product.

For this application, the MoSCoW Method is the prioritization technique used to determine the importance of each item. The items are labeled with the level of importance based on the customer and their requirements.

The acronym stands for:

  • Must have: items that should be finished and delivered to the customer as fast as possible. If by the end of the project, a "Must Have" item is not finished, the project has failed to be completed in time.

  • Should have: items that can be equally important with a "Must Have" item, but there is enough time to finish them at a later date or there may be other ways to satisfy the same requirement.

  • Could have: items that are desirable by the customer but are not essential features of the product developed. These are usually included if enough resources (time, development cost etc.) can be spared.

  • Won't have: items that are of least importance during the development of the product. The customer has concluded that these items are not needed or they need further reconsideration as they may not be appropriate at that time.

Some notes: Priority can be given to any of the items created, whether they are epics or tasks, as they help the team prioritize in any case, but only the product backlog items are sorted and not their children.

Item Effort

This is the effort of an item.

The effort is how difficult an item is, according to the team. It is a way of expressing how much time, the rate of difficulty, the technical expertise etc. required to implement a product backlog item or any other piece of work.

It is calculated amongst the team, during the discussion with the product owner and the backlog refinement phase. It is done during the planning poker part, where each team member writes down on a piece of paper, their personal estimate of how difficult they believe the item is. After revealing the cards and discussing with the other team members, the team agrees to an average effort required by the item.

What matters is not how much the number representing the item is, but the ratio in comparison with the other items. If for example an item has an effort of 1, and another item has an effort of 3, it means that the latter requires triple the amount of work of the first.

Agile enthusiasts may not use the effort system, but one of their own. It depends on the team and their ability to understand and represent effort in their own way. For example, a team may use as a metric, the number of cookies that are consumed during the development and implementation of an item.

In this application, the effort is estimated using the decimal system. 1 is for the lowest effort, whereas 10 is for the highest effort. Any more or less than that is not possible, and the system will reset it to one of the edge numbers.

Some notes:

Epics and stories have a different meaning of effort compared to tasks and bugs. The epics and stories have an "Estimated Effort" which is estimated in case tasks have not yet been created for each epic and its children stories. One of the reasons this happens, is because the development team needs to realise the amount of work they believe an epic or a story will need in order to be finished. On the other hand, tasks and bugs have a clearer definition of "Effort", where a more pragmatic estimation is done as the product backlog item has now been broken down into the lowest level possible. Epics and stories also have a second attribute, which is the "Total Effort" of all the tasks and bugs that are included within them.

Item Assignee

This is the assignee of the item.

The product owner has the ability to assign someone who will be responsible for the completion of the item.

If it is a task, he or she will be able to move it in the task board, and whenever they are done, they will be able to assign it to somebody else.

In Agile, and especially Scrum, practices such as pair programming and daily collaboration and discussion are encouraged, and the team should always work together towards the completion of an item.

Some notes: The product owner can modify an item (including deletion), whereas assignees can only move tasks in the task board or assign new team members to work on the item.

Sprint Task Board

This is the task board of the sprint.

Through the task board, the sprint backlog is made available for all the team members to see and act upon.

There are 4 columns where the team members constantly move tasks and bugs by updating their status.

The task board helps the team keep track of all the tasks that have not yet started, are close to completion or have been finished.

A task board may also include notes, but this application does not support this functionality.

To Do

This is the “To Do” list.

In the “To Do” list, all the items that have been added to the sprint backlog prior the sprint execution, will be contained.

These are the items that the development team will have to work on during the sprint execution.

They have not yet tarted, and each team member is assigned to a number of these. When he or she is ready to start working on them, they will have to move it to the “In Progress” list.

In Progress

This is the “In Progress” list.

In the “In Progress” list, all the items that the development team is currently working on, are contained.

They usually come here from the “To Do” list, but they can also be returned back from the “For Review” list.

It is possible that the assignee might change during this phase of the task.

For Review

This is the “For Review” list.

In the “For Review” list, all the items that have finished being developed and require further actions such as testing, are included here.

The developer will usually assign the task to somebody who will test the code, or to somebody who will review it.

In case the item fails to pass the tests or is not finished for whatever reason, it will return to the “To Do” list or to the “In Progress” list, depending on the team and what they have agreed upon such a case. Some teams may also have secondary tasks that concern the testing of a task. Depending on the outcome of the secondary tasks, the original task will be either moved back, or completed.

Done

This is the “Done” list.

In the “Done” list, all the items that the development team has finished working on, are contained.

In order for an item to qualify as “Done”, they will need to have passed the acceptance criteria the team has set for each item individually or collectively. Example criteria may include:

  • Testing
  • Refactoring
  • Ability to ship

The velocity of the sprint is calculated by summing up all the effort of the finished tasks and bugs.

Some notes: If a task is near completion (but has not been completed yet), it should not be moved to the “To Do” list. It should remain at its current state and added for completion in the next sprint. Avoiding deception is more important than velocity and numbers, as the former helps the team in having a clear view of the product.

Burndown Chart

This is the Burndown Chart.

The burndown chart is a graphical representation of the amount of work left to do in comparison with the duration of the sprint.

It is being used by the development team in order to track their progress and evaluate the outcome of the sprint. Some examples of how it helps the team include:

  • If work finishes early, it might mean that not serious commitment is done by the team.
  • If the amount of work is above the ideal line, it might mean that a task includes way too more work than expected.
  • If the amount of work is represented by a sharp drop, it might mean that the effort for the items completed might not have been estimated accurately.

In this application, the vertical axis describes the effort and the horizontal axis shows the days until the competition of the sprint.

The burndown chart starts from the total effort, and every day, the amount of effort burned by the team is calculated.

Ideally there will be a steep downward slope which leads to 0 effort remaining. When it reaches 0, it means that all the items contained into that sprint have been completed and the sprint should be finished.

The ideal remaining is calculated by the following formula:

  • (Total Sprint Effort) / (Duration of Sprint)

Sprint History of Tasks Done

This is the Sprint History of Tasks Done.

This table shows the items that have been finished in a particular date.

This helps the team track when an item was finished.

If the user clicks on the item, he will be able to view the details of it.

Product Burndown Chart

This is the Product Burndown Chart.

The product burndown chart is a graphical representation of the amount of work left to do in comparison with the number of sprints forecasted by the product owner or the team.

Depending on the number of previous sprints and the velocity of each sprint, the forecast will constantly change and become more and more accurate.

Studying this chart, helps the product owner in re-adjusting deadlines or changing the priority of product backlog items and user stories.

In this application, the vertical axis describes the total effort of the product burned each sprint, and the horizontal axis shows the number of sprints that have been executed. It also includes the current sprint, if there is one currently executing.

Ideally, there will be a steep downward slope which leads to 0 effort remaining. When it reaches 0, it means that all the items in the product backlog have been finished, and the product has been completed.

The ideal remaining is calculated by the following formula:

  • (Project Total Effort) / (Estimated Sprints Needed)

Some notes: Some teams may completely ignore the idea of a product burndown chart. This chart is not necessary, but the developer has opted to leave it in the application, for educational purposes.

Sprint History

This is the Sprint History.

The sprint history shows all the previous sprints that have been executed and all the items that were included in a particular sprint, along with the status they had, when the sprint deadline was reached.

This table helps the product owner and the development team track all the tasks that have been part of a sprint.

This table is the only way to access and view tasks, after they have been finished.

If the user clicks on the item, he will be able to view the details of it.

If the user clicks on the sprint row, he will be redirected to the task board page of that sprint, where they can see in detail the burndown chart and the date a particular item was finished.

Scrum Roles

These are the roles of Scrum.

In Agile and especially in Scrum, the sense of hierarchy is very limited and almost non-existent. People work in a team and the work done, is represented by the team as a whole. The Scrum team is self-organizing and does not depend on third parties or is directed by others.

Each role in Scrum, represents a set of characteristics and responsibilities. These responsibilities do not affect each other and do not disrupt the responsibilities of other roles. Any individuals or entities that are not part of the following Scrum roles but affect the execution of a Scrum project, are considered a major disruption and should be dealt with immediately:

The Product Owner

The product owner is the voice of the customers, as he is the one that communicates with them. He is responsible for understanding the customer needs and passing them to the rest of the Scrum team.

Some of his most important responsibilities include:

  • Creating the Product Backlog.
  • Prioritizing the Product Backlog to best express the most important customer needs.
  • Making sure the Scrum team understands all the items in the Product Backlog to the level that is needed.

The product owner can only be an individual, but he can represent the voice of many. If someone wants to re-prioritize an item, then they should address the product owner.

The product owner’s decisions should be respected for the normal execution of the project.

The Development Team

The development team is the set of developers or experts that are responsible for delivering a potentially shippable product by the end of each sprint.

The development team is also responsible for the creation of the Sprint Backlog. They are the ones that organize and manage what they will be working on each sprint.

Some of their characteristics include:

  • They are self-managing and only they should know how to organize the work each increment in order to ship a product feature.
  • They are independent. The team is supposed to have the needed set of skills in order to successfully ship the product.
  • They have no titles. It is in the culture of Scrum for development teams to not have any titles regarding their expertise.
  • There are no sub-teams. Each team member is supposed to have the knowledge in all the phases of development, including testing, operations etc.
  • Accountability is represented by the team as a whole. The team is responsible for shipping a feature successfully, not the individuals that are included in the team.

The development team is supposed to be between 3 and 9 members. Above 3 so that they manage to deliver enough work each sprint, and below 9 so that coordination and self-organization remains effective. The rest of the roles are not included in these numbers unless a team member has more than one role.

The Scrum Master

The Scrum master is the one that promotes the culture of Scrum and supports the theory behind it. He is responsible for making sure that the Scrum framework is followed by the Scrum team as it is supposed to be, and he is the one that helps everyone understand the practices, rules etc. of Scrum.

The Scrum Master is a servant-leader that services the product owner, the development team as well as the organization.

He serves the product owner with the following ways:

  • Helping in efficiently organizing the product backlog.
  • Helping the Scrum team in understanding the goals, scope and product domain.
  • Making sure agility is understood and practiced.

He serves the development team with the following ways:

  • Teaching the development team the value of self-organization and self-management.
  • Helping the development team in creating high-value products by ensuring criteria such as INVEST are followed.
  • Removing any impediments or anything that obstructs the development team from working efficiently.

He serves the organization with the following ways:

  • Making sure Scrum is followed where it can be applied.
  • Helping employees and stakeholders understand the values of Scrum and how important they are for the proper functioning of the Scrum team.
  • Causing change that increases the results delivered by the Scrum team.

In this application, these are the functions of each role:

Product Owner:

  • Create/Delete a project.
  • Create/Delete an item.
  • Edit Epics and Stories.

Development Team:

  • Create/Delete Tasks and Bugs.
  • Edit Tasks and Bugs.
  • Move item in the task board (if item is assigned to self).
  • Change assignee of item (if item is assigned to self).

Scrum Master:

  • Start/Finish sprint.
  • Move item to sprint.
  • Remove item from sprint.
  • Change assignee of item.
  • Everything that the other roles can do.

Some notes: If the team does not wish to have separated functionalities but would like to have all functionalities in one role, then they should choose the admin role for each user.

Add new Task

This is the add new task functionality.

The stories and tasks that are included into the active sprint backlog are the result of the discussions that took place during the backlog refinement and the sprint planning meetings.

In the former, one of its key characteristics is the fact that the Scrum team ‘refines’ the items that are included in the product backlog. This means that the development team, in discussion with the product owner break the epics into stories and the stories into tasks and bugs.

In the latter, which is the sprint planning, the development team talks about the items that they will work on during the execution of this sprint. This happens at the beginning of the sprint and some of the key points made, include:

  • What is the goal of the sprint?
  • What are the items included in the product backlog that fulfill this goal?
  • Who will be able to work on this sprint and what is the team capacity based on availability?
  • What are the items that will be included based on the capacity?
  • How sure is the team they are going to reach the sprint goal?

Lastly, if needed, the team will continue breaking down the stories into more tasks. However, during the decomposition of the items into smaller ones, it is important for the team to keep the planning limited. Usually, discovering 60%-80% of the tasks that the team will work on during the execution of the sprint, is enough to get going. It is exhausting for the team to find all the tasks that will need to be worked on, and it is unavoidable to have unexpected tasks and bugs that will be revealed only during the execution of the sprint.

Therefore, the developer team is given the opportunity to add tasks or bugs during the execution of the sprint. These items will consist either of the unexpected work that the team will have to do in order to complete the goal or will be the result of the re-estimation of some of the stories the developers will do during the sprint.

Some notes: Once a task is created, it is important for the user to make sure that the task will not be changed. The ability to edit items is only given to the product owner and is only intended during the backlog refinement meeting. If the user wishes to edit any of the tasks, to re-estimate for example, then a user with the product owner or admin role will be able to do so.

Commect Section

This is the comment section.

The comment section is used in order to share information between the team and as a way for the current assignee to leave documentation for the next assignee.

An assignee will usually post a comment after they are done working on the task in order to help the next assignee by either writing short sentences such as descriptions of the work done, problems they might have encountered and need help with or simply reminders for the team to see.

In Agile and especially Scrum, collaboration between the team members is encouraged and a comment section contributes to that vision.
